# JWBuddy：纪检监察智能体客户端平台 — 设计文档

- **日期**: 2026-06-23
- **项目**: JWBuddy (jwbuddy)
- **状态**: 设计确认稿

---

## 1. 概述

### 1.1 项目定位

为纪检监察部门打造一套类似 Claude Code / Codex / WorkBuddy 的**通用智能体客户端平台**，支持：

- **自然语言驱动的数据分析** — 纪检人员用对话即可查询数据、发现异常、生成图表报告
- **插件化扩展体系** — 所有业务能力（养老、三资、招投标、教育、四风、信访等）以 Skill/MCP/Tool 形式接入
- **多形态客户端** — Desktop（主力工作台）、CLI（批量自动化）、App（移动查阅）
- **私有化部署** — 内网 910B 集群推理 + 政务云大模型双轨并行

### 1.2 目标用户

| 阶段 | 用户范围 | 规模 |
|------|---------|------|
| MVP 试点 | 某科室 | ~10 人 |
| 推广阶段 | 全省 21 市州纪检部门 | ~1000 人 |

### 1.3 部署环境

| 环境 | 说明 | 优先级 |
|------|------|--------|
| 政务外网 | 非敏感数据，政务云 LLM | 第一优先级 |
| 内网 | 涉密数据，910B 私有推理 | 第二优先级 |

### 1.4 时间预期

- **MVP 开发周期**: 1-2 个月
- **试点**: 第 7-8 周

### 2.2 核心设计原则

| 原则 | 说明 |
|------|------|
| **插件化优先** | 所有业务能力 = Skill/MCP/Tool，底座不耦合业务 |
| **国产化全栈** | 910B + DeepSeek/Qwen + 国产数据库适配 |
| **内外网分离** | 核心逻辑同构，数据源和模型分别适配 |
| **安全审计** | 所有 Agent 操作可追溯、可回放、可审批 |

## 3. Agent 核心与多智能体运行时

### 3.1 架构

```
输入 → Planner → 分解任务
         ├── No  → Executor 直接执行
         └── Yes → Orchestrator
                    ├─ Worker Agent 1
                    ├─ Worker Agent 2
                    └─ Worker Agent N
                    → 结果聚合 → 输出
```

### 3.2 技术选型

| 组件 | 选择 | 理由 |
|------|------|------|
| Agent 框架基底 | LangChain | 成熟的 Agent/Tool 生态 |
| 多 Agent 编排 | LangGraph | 有向图调度，支持复杂工作流 |
| Schema 定义 | Pydantic | 工具/Skill Schema 校验 |
| 会话状态 | Redis | 支持断连恢复 |
| 持久化 | SQLAlchemy | 审计日志、会话存储 |
| Agent 模式 | ReAct | 思考-行动-观察循环，适合数据分析 |

### 3.3 Agent 实例结构

每个 Agent 实例包含：
- **System Prompt**: 角色定义 + 权限范围
- **Conversation Memory**: 会话短期记忆
- **Available Tools**: 当前可用工具列表
- **Skills**: 已加载的技能包
- **MCP Clients**: 连接的外部服务

### 3.4 典型工作流

用户提问 → Planner 分解为子任务 → Worker 并行执行 → 汇总 → LLM 组织回答

## 4. 扩展体系: Skill / MCP / Tool

### 4.1 三层扩展架构

```
用户交互层 ──→ Skills 层 (编排层，面向业务场景)
                  ├─ 养老资金分析 skill
                  ├─ 招投标围标分析 skill
                  └─ 信访分析 skill
                  ↓
             Tools / MCP 层 (执行层，原子能力)
                  ├─ sql_query / vector_search
                  ├─ document_parse / chart_generate
                  └─ MCP Server (外部数据源桥接)
```

### 4.2 Skill 定义规范

每个 Skill 是一个目录，包含 `skill.yaml` 声明 + Python 实现：

```yaml
name: pension-analysis
description: 养老资金异常分析
triggers:
  - "养老*资金*异常"
  - "养老金*冒领"
tools:
  - name: query_pension_data
    description: 查询养老资金发放数据
    parameters:
      type: object
      properties:
        id_card: { type: string }
        time_range: { type: string }
workflow:
  - step: 查询数据 → query_pension_data
  - step: 异常检测 → detect_anomaly
  - step: 生成报告 → LLM summarize
```

### 4.3 MCP 协议桥接

Agent Runtime ↔ MCP Protocol (Stdio/SSE) ↔ MCP Server (SQL/文件/API/向量库/OCR)

### 4.4 内置 Tool（MVP 可用）

| 类别 | Tool | 说明 |
|------|------|------|
| 数据 | `sql_query` | 自然语言 → SQL → 查询结构化数据 |
| 数据 | `vector_search` | 语义搜索（法规/案例库） |
| 文档 | `document_parse` | PDF/Word/图片 OCR + 结构化 |
| 分析 | `data_analyze` | Pandas 自动统计分析 |
| 可视化 | `chart_generate` | 自动生成 ECharts 图表 |
| 报告 | `report_generate` | 生成分析报告初稿 |

## 5. 数据分析引擎

### 5.1 MVP 核心能力

自然语言驱动的数据查询与分析：

```
用户提问 → NL → SQL 转换层 (LLM + Schema 感知 + 领域术语映射 + 安全过滤)
        → 数据查询执行层 (SQL 校验 → 安全沙箱 → 执行 → 结果封装)
        → 后处理与展示层 (LLM 分析 → 自动判断展示方式: 表格/图表/关系图)
```

### 5.2 多轮对话分析

支持连续追问，保持上下文，逐步深入分析。

### 5.3 非结构化数据处理

PDF/Word/图片 → OCR → 文本提取 → Embedding → 向量库 → 语义检索(RAG) → LLM分析

### 5.4 MVP 范围

| 模块 | 交付 |
|------|------|
| NL→SQL 引擎 | 对接 1-2 个业务数据库，支持对话式查询 |
| 可视化展示 | 自动判断图表类型，ECharts 渲染 |
| 报表生成 | 数据分析报告初稿 + 导出 Word |
| 文档解析 | PDF/Word 上传统一检索 |

## 6. 客户端架构

### 6.1 Desktop（MVP 主力）

- **壳**: Tauri 2.x (Rust)
- **UI**: React + Ant Design + ECharts
- **后端通信**: HTTP/WebSocket ↔ Python Agent Sidecar
- **功能**: 对话面板、数据表格、可视化图表、文档预览、历史会话、Skill管理器、MCP连接状态

### 6.2 CLI

- **框架**: Click + Rich
- **功能**: 命令行查询、数据导出、Skill/MCP/数据源管理

### 6.3 App（MVP 轻量版）

- **框架**: Flutter
- **功能**: 会话列表同步、消息通知、数据看板(只读)、快速语音查询

## 7. LLM 集成

### 7.1 双轨推理

| 场景 | 路由目标 |
|------|---------|
| 日常辅助 | 内网 DeepSeek |
| SQL 生成 | 内网 Qwen |
| 复杂推理 | 内网 DeepSeek |
| 非敏感通用 | 政务云 |
| 涉密数据 | 内网 ONLY |

### 7.2 向量检索

BGE-M3 (国产 Embedding, 910B推理) → Milvus 向量库

### 7.3 路由策略

基于请求敏感度 + 推理复杂度 + 负载均衡的自动路由。

## 8. 部署架构

### 8.1 双环境

| 环境 | 组件 | 数据 |
|------|------|------|
| 政务外网 | Agent 服务 + 政务云 LLM + App | 非敏感数据 |
| 内网 | Agent 服务 + 910B 集群 + Milvus + PostgreSQL | 涉密数据 |
| 内外网之间 | 物理隔离 / 光闸 | 不互通 |

### 8.2 安全设计

| 关注点 | 方案 |
|--------|------|
| 数据隔离 | 内网数据不出内网 |
| 审计日志 | 全操作记录（谁、时间、内容、结果） |
| SQL 安全 | 只读查询，禁止 DDL/DML，自动脱敏 |
| 文件安全 | 沙箱扫描，处理即删 |
| 身份认证 | OAuth2 + LDAP |
| 权限控制 | Skill 级 + Tool 级 + 数据源级 |
| 模型安全 | 内网私有部署 |

### 8.3 MVP 部署规格（10人试点）

| 组件 | 规格 |
|------|------|
| Agent 服务 | 8C 32G × 2 |
| 910B 推理 | 共享已有集群 |
| PostgreSQL | 4C 16G 500G |
| Milvus | 4C 16G |
| Desktop 客户端 | 普通办公 PC |
